# Skill: Decoupling and Splitting Resolvers in Federated GraphQL (Golang + gqlgen)

This guide documents the steps and best practices to move, split, and decouple hand-written resolvers from gqlgen's default `schema.resolvers.go` file into a dedicated package and structure.

---

## Architecture Overview

By default, `gqlgen` generates resolver stubs inside the `graph/` package under `schema.resolvers.go`. To decouple resolvers for better maintainability and clean domain boundaries, you can structure your subgraph module like this:

```text
/customer-subgraph
├── graph/               # GraphQL execution package (generated code, schemas, and models)
│   ├── generated.go     # Generated by gqlgen (DO NOT EDIT)
│   └── schema.graphqls
├── resolvers/           # Hand-written resolver implementations
│   ├── resolver.go      # Defines the root Resolver struct and injected dependencies
│   ├── bet.resolvers.go # Split resolvers by domain
│   ├── query.resolvers.go
│   └── ...
├── server.go            # Subgraph entry point
└── gqlgen.yml           # gqlgen configuration
```

---

## Implementation Steps

### Step 1: Create a Dedicated `resolvers/` Package

1. Create a `resolvers/` directory inside your subgraph.
2. Create the root resolver file: `resolvers/resolver.go`. Define your root `Resolver` struct with its dependencies (e.g., mock database/API layers).

   ```go
   package resolvers

   import "customer-subgraph/datasources"

   type Resolver struct {
       CustomerAPI *datasources.MockAPI
   }
   ```

### Step 2: Implement Domain-Specific Resolver Files

For each GraphQL resolver interface (e.g., `BetResolver`, `QueryResolver`), create a corresponding Go file in the `resolvers/` package (e.g., `resolvers/bet.resolvers.go`).

> [!IMPORTANT]
> Because `resolvers` is a separate package from `graph` where the gqlgen interface types are defined, you must prefix resolver interfaces with `graph.` (e.g., `graph.BetResolver` instead of `BetResolver`).

Example `resolvers/bet.resolvers.go`:

```go
package resolvers

import (
    "context"
    "customer-subgraph/graph"
    "customer-subgraph/graph/model"
    "customer-subgraph/resolvers/entities"
)

// Bet returns graph.BetResolver implementation.
func (r *Resolver) Bet() graph.BetResolver { return &betResolver{r} }

type betResolver struct{ *Resolver }

// Game is the resolver for the game field.
func (r *betResolver) Game(ctx context.Context, obj *model.Bet) (*model.Game, error) {
    return entities.BetGame(ctx, r.CustomerAPI, obj)
}

// Amount is the resolver for the amount field.
func (r *betResolver) Amount(ctx context.Context, obj *model.Bet) (float64, error) {
    return obj.Amount, nil
}
```

### Step 3: De-conflict gqlgen Configuration (`gqlgen.yml`)

Comment out the `resolver:` generation configuration block in `gqlgen.yml`. This prevents gqlgen from generating/overwriting resolver stubs (e.g., `schema.resolvers.go`) inside the packages and causing duplicate definition conflicts during build time.

```yaml
# gqlgen.yml

# Comment this block to manage resolvers manually:
# resolver:
#   dir: resolvers
#   package: resolvers
#   filename: resolvers/schema.resolvers.go
```

Remove the obsolete `resolver.go` and `schema.resolvers.go` files from the `graph/` package once the resolvers have been successfully migrated to the `resolvers/` package.

### Step 4: Update the Server Entry Point (`server.go`)

Update `server.go` to import the new `resolvers` package and inject `&resolvers.Resolver{}` into the executable schema config instead of the old `graph.Resolver`.

```go
package main

import (
    "customer-subgraph/graph"
    "customer-subgraph/resolvers"
    // ... other imports
)

func main() {
    // ... setup code
    customerAPI := datasources.NewMockAPI()
    srv := handler.NewDefaultServer(graph.NewExecutableSchema(graph.Config{
        Resolvers: &resolvers.Resolver{
            CustomerAPI: customerAPI,
        },
    }))
    // ... start server
}
```

### Step 5: Regenerate & Verify

Run code generation and compilation to verify that the split resolver setup is correct:

```bash
# Generate gqlgen code
cd customer-subgraph && go run github.com/99designs/gqlgen generate

# Compile package to verify absence of duplicate definitions
go build ./customer-subgraph/...
```
